Core Principle of REST:
    (1) client AND SERVER ARE INDEPENDANT entities as they have nothing to do with each other like server does not care hove the 
    data is presented in frontend etc 
            client handels the interface, server handles the data and logic 
            communication only occurs through standardized requests and responces.

    (2) Statelessness-each request is self contained, carrying all the information the server needs. The server does not store 
    client context between requets/don't remember the request-each request is brand new.

    (3) Cacheability-responces must indicates wheter they are cacheable. Proper caching improves performance, reduces load and 
    enhance network efficiency-by sending fever request to the server.
        -> data that changes frequently should not be cached like live stock prices etc
    
    (4) Layered system - the server can have hierarchial layers(load balncer, cache, security, other intermediaries). the client 
    interact only with layer it sees, unaware of others, improving scalability and security(the client doesn't needa know where 
    the rquest goes, i tonly cares about the data recieved).

    (5) Code on demand - optionally server can send excutable code to extend client functionality. this is optional, not requires 
    for rest and often avoided.

    (6) Uniform Interface - Heart of the REST
        a) resource identification in URLs - every resource has a unique url/endpoint.
            i.e: post/ -> all posts 
                 posts/1/ -> post with id 1
                
        b) standard HTTP methods - use HTTp verbs/methods consistently: GET, POST, PUT, PATCH, DELETE.


    urls represents resources(nouns), NOT actions(verbs)
        GET/getAllPosts/  ==> incorrect
        correct => noun + HTTP method ==> GET/posts/


        c) Self-Descriptive Messages - each request and responce contains all needed info.

        d) HATEOAS - HYPERMEDIS AS the engine of applications state - responces can include link to guide the client on next actions
         in simple the server can include links in the responce, and the user can understand what acrions are possible here  


Server-Client communication - so the client send a http request through a url/endpoint based on what action he wanna take, on the serverb 
defined the endpoints and whenever it hitted the server will perform the specified action and will return the data in responce
